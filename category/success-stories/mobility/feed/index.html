Porting to the New Generation Steering Column Control Module

Porting to the New Generation Steering Column Control Module

CASE STUDY

  • MOBILITY

Business Overview

The objective of the project is to port from microcontroller Cypress Traveo 2 (100 pin) to Cypress Traveo 2 (80 pin). The client shared the  Existing configuration for Cypress Traveo 2 – 100 pin. Our team needed to incorporate all the changes that occurred, as a result of this porting. 

Challenge

The microcontroller was modified and the client needed to incorporate all the changes that occurred as a result of this porting. Configuration, Validation and Autocode generation of numerous modules of AUTOSAR Layers needed to be performed.

Solution

The Steering Column Control Module (SCCM) is an ECU that controls steering in the switches. The scope of the project involved  porting from  , Cyprus Traveo 2- a 100 pin microcontroller, to  Cyprus Traveo -2 ,80 pin microcontroller. The customer shared the Existing configuration for Cypress Traveo 2 – 100 pin.  

Configuration, Validation and Autocode generation of the modules of the following AUTOSAR layers needed to be performed.  

  • MCAL layer:  We identified ten modules in this layer that needed to be configured. These are Gpt, Mcu, Can, Lin, Spi, ADC, Dio Port, Pwm and Fls.
  • ECU Abstraction Layer- Configuration changes are done only in CanIf module 
  • System Service Layer- Based on the new CAN DBC and CDD File, we configured the COM stack modules such as COM, PduR and diagnostic modules like DEM , DCM.The new DBC file for communication stack was updated to facilitate BSW communication

For the modules with no delta changes, auto code generation was used. 

We also handled the generation and maintenance of The Run Time Environment (RTE) Layer and Updated AUTOSAR Interfaces to application components 

Since the existing project was re-used, numerous modules were auto code generated and the auto generated code was added to the existing software workspace. 

Based on the CAN DBC file provided by the customer, AUTOSAR modules like Canif, PduR, and Com were configured. Similarly, Configured diagnostic modules such as dem, dcm, and det  based on new CDD files from the customer 

Value Delivered

There was a need to port from one microcontroller to another to ensure  Hardware Availability and cost-effectiveness. When confronted with this challenge, the customer approached us. With our well-established execution strategy, we were able to successfully handle the software scope of the project

System Basis Chip (SBC) Upgrade of Steering Column Control Module (SCCM)

System Basis Chip (SBC) Upgrade of Steering Column Control Module (SCCM)

CASE STUDY

  • MOBILITY

Business Overview

Due to the global shortage of SBC Chipsets, The Client had a requirement to redesign hardware for their OEM client by replacing the SBC Chip in their Steering column control Module. Currently, the SBC Chipset used is TLE926QX, which needed to be replaced with UJA1169ATK. During this replacement, the existing functionalities needed to remain operational. 

Challenge

The Client’s First scope was to work for the first variant-” Steering Column Module U625/CX482 with Heated Steering Wheel with Tilt Tele”. 

They approached our team of experts with the following scope of development: 

 

  • The current SBC Chipset TLE926QX needed to be replaced by a new SBC UJA1169ATK 
  • To modify the existing SBC driver for the new SBC chipset.  
  • To modify the new CAN transceiver and the LIN transceiver and implement changes in SBC HS Signals due to the removal of the existing SBC chip. This change, in turn, will affect the bootloader and the application. 

Solution

The team identified from the System Architecture and the original code shared by the client, all the modules and functions that needed to be ported. 

  • The solution involved changes to both the Application Software and Bootloader Software 
  • The new hardware incorporated CAN Transceiver Integration and LIN Transceiver Integration 
  • In the new implementation MCU pin is used to turn ON/OFF the LIN phy. 
  • LIN Transceiver Integration: In the old design,  LIN is enabled by the SBC pin. In the current design, the extra microcontroller pin is used to enable LIN phy.  
  • The replacement of the current SBC also required the SBC CDD Driver to be ported.  
  • The Changes for SBC HS signal Handling- Understanding & Removal of Existing HS Signal Code.  
  • Added Two Additional Input/output Pins for LIN Enabling and Heated Steering Wheel (HSW) Low side enabling 
  • In the previous design, the SBC Chip itself handled the HS signals. One of the pins in SBC Chip enabled and disabled the HSW. In the current design, one of the GPIO pins in MCU is responsible for enabling and disabling HSW.  
  • Removal of external watchdog functionality. Since the new SBC uses Internal Watchdog which can be used in the autonomous mode. 
  • Diagnostics: Analysis of changes due to the unavailability of some modes of SBC. 

 

Value Delivered

With its strong expertise in vehicle electronics, our team of experts was able to replace the existing SBC with a new SBC and retain all the existing functionalities by redesigning the hardware.

Platform Support for Next-Gen Body Control ECMs

CASE STUDY

  • MOBILITY

Business Overview

The client is a global manufacturer of automotive components, a Tier 1 supplier to major OEMs around the world. They were entrusted with developing numerous modules within the major ECU’s of an OEM’s set of new generation vehicles. The main ECUs involved were the Body control module (BCM) and Enhanced lighting module (ELM) of the vehicle. 

BCM is the master module which handles the major cabin controls of the vehicle whereas ELM controls mainly the lighting of the vehicle. A BCM in automotive can perform a wide range of functions such as Passive Entry Passive Start (PEPS), Remote Keyless Entry (RKE), Climate Control System, Power Windows etc 

We got on board the project due to our extensive experience in Vehicle Electronics. Our team of experts extended their support in designing, developing, and testing the Complex Device Drivers for their Body control ECMs. 

We worked on the device driver layer and the Hardware abstraction layer (HAL) of these ECUs. The Hardware abstraction layer (HAL) acts as an interfacing layer between the hardware and the rest of the ECU software.  ADC, CAN, DIO, FLS, GCL, ICU, LIN, MCU, PORT, PWM, SPI, WDG are some of the modules that we develop.   

In Turn, the application layer, which is responsible to use the released work products, was developed by the OEM. 

Challenge

Lack of simulation setup

Software Architecture 

Solution

The Application Layer was developed by the OEM directly. Application has many modules that enable power window opening, remote keyless entry etc.  

  • Developed and Designed Test cases of the Test Modules and other modules in Complex Drivers 
  • MCAL Porting: During the transition to Gen 1 to Gen 2 we enabled the Configuration of modules within the MCAL driver 
  • Unit Testing , module testing and comprehensive Checks of the whole workspace 
  • Reduced manual effort of testing by automating the test cases using Cantest automation 
  • Automated the modules developed by client

Implementation of 2 Speed Torque on Demand (TOD) Transfer Case system

Implementation of 2 Speed Torque on Demand (TOD) Transfer Case system

CASE STUDY

  • MOBILITY

Business Overview

The Client required a 2 Speed Torque On Demand (TOD) Transfer Case system using the existing hardware for Transfer case Control Unit(TCU). The transfer case needed to transition the vehicle from 2WD to 4WD and vice versa effortlessly. 

Challenge

The client had already developed existing technology but the software was binary. Accordingly, NeST redeveloped the existing benchmark technology on the basis of the requirements provided by the customer. 

Solution

The transfer case needed to transition from 2WD to 4WD and vice versa effortlessly.  

The four different modes of the transfer case are as follows: 

  1. 2H mode
  2. Auto mode
  3. 4H Mode (4WD Lock)
  4. 4L Mode (4WD Low Range)

These modes will be selected based on the vehicle mode provided by the ATS or ESP ECU of the vehicle. The transfer case ECU will change the modes based on different geographical terrains. It is able to control the torque by controlling the Clutch actuator and in turn, the percentage of this clutch control is determined with the help of different CAN signals like Wheel speed, Throttle position, Steering angle etc.  (TOD Transfer case will be a node in the Vehicle CAN network so this module will follow CAN network specification.) 

In addition, it monitors the temperature and will initiate temperature shutdown if the temperature increases. Temperature monitoring and Thermal Shutdown Strategy shall be carried over from a former project with the client. 

Features:

  • Network Failure Diagnosis & DTC Logging 
  • Bootloader and firmware update using UDS protocol 
  • Thermal Model Interface- A thirdThird Party Interface that will take different CAN inputs for temperature monitoring and initiate temperature shutdown if necessary. 
  • Transmission Control Manager: Responsible for evaluating the conditions required to initiate a shift. It identifies the current shift actuator position and the shift input, compares both and initiates a shift.  
  • Support for runtime calibration of parameters through CAN Calibration protocol. 
  • Traction and Slip Control 
  • Shift Handler: Gets information from the Transmission control manager on whether to initiate a shift or not. Shift System is responsible for moving the Shift Actuator into the required position based on a Shift System Mode Input. 
  • Supports the software reprogramming through CAN using UDS protocol 
  • System Diagnostics: Variant Code Checking, Synchronisation actuator diagnostics, Shift motor diagnostics, Motor position encoder diagnostic

Value Delivered

  • Allows for smooth transition between different TCCU modes  
  • Torque Management 
  • Slip Management 
  • Support for calibration update using CCP 
  • Firmware Update using UDS Protocol

Encrypted Secure Boot for DDS Rotary Knob Module

Encrypted Secure Boot for DDS Rotary Knob Module

CASE STUDY

  • MOBILITY

Business Overview

With the ever-increasing complexity of vehicles, cybersecurity is a primary concern in vehicle design. The Client needed to implement a secure boot feature for a DDS Rotary Knob module for their passenger car. The secure boot feature ensures that the code executed on the product is trusted and authentic.

Challenge

One of the key challenges, in the rising threat of cyber-attack is to ensure secure automotive embedded systems.

Solution

The scope of the undertaking is the implementation of a secure boot feature in the DDS Rotary Knob Module.

The hardware for this project is based on i.MX-RT 1176 platform and for this family of processors; secure boot is implemented with the help of AHAB.

The solution was implemented in two phases which involved Manual Code Signing using NXP  SPT Tools and Code Signing using Scripts and CLI Tools. 

Features:

  • HAB Secure Boot: Asymmetric Algorithm to perform Authentication Process

  • Encrypted Boot 

  • Authenticated image creation using Scripts 
  • Key fusing using scripts 
  • Manual Code Signing using NXP SPT Tools 
  • Code signing using scripts and command line tools 
  • Custom Bootloader and Application Firmware bin file creation 
  • Generation of Keys, signing of images, and encryption of images used in secure boot 
  • Generation of signed encrypted bootable image

Value Delivered

  • Secure Boot ensures that unauthorized, tampered, or modified images are not loaded during the Boot sequence
  • On detection of a modified software component, secure boot is able to perform active countermeasures 
  • Ensures protection of Intellectual Property (IP).
